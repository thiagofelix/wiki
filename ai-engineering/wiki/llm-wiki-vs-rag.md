# LLM Wiki vs RAG

The [[llm-knowledge-bases|LLM Knowledge Base]] (wiki) approach and traditional Retrieval-Augmented Generation (RAG) are two different strategies for giving AI access to your knowledge. The wiki approach uses organized markdown files with indexes and backlinks; RAG uses vector embeddings and similarity search. Each has strengths at different scales.

## Comparison

| Dimension | LLM Wiki | Semantic Search RAG |
|-----------|----------|-------------------|
| **Retrieval** | Reads indexes, follows backlinks | Vector similarity search |
| **Understanding** | Deep — linked relationships | Shallow — "these chunks seem similar" |
| **Infrastructure** | Just markdown files | Embedding model + vector DB + chunking pipeline |
| **Cost** | Tokens only | Ongoing compute + storage |
| **Maintenance** | Run a lint, add articles | Re-embed when things change |
| **Scale limit** | ~Hundreds of pages | Millions of documents |
| **Setup time** | 5 minutes | Hours to days |

Source: [[source-herk-llm-wiki-setup|Nate Herk's]] comparison chart generated by Claude Code.

## Where Wiki Wins

- **Personal knowledge bases** — At ~100 articles / ~400K words, the LLM navigates indexes and finds data without vector search
- **Relationship depth** — Backlinks encode actual semantic relationships, not just embedding proximity
- **Zero infrastructure** — No database, no embeddings, no chunking pipeline. Just folders and files
- **Cost** — Effectively free beyond token usage
- **Interoperability** — Plain files work with any tool ([[obsidian]], [[claude-code]], VS Code, grep)
- **Token efficiency** — One user reported 95% token reduction vs. querying raw files directly

## Where RAG Wins

- **Enterprise scale** — Millions of documents require vector search; file-based navigation breaks down
- **Real-time ingestion** — Vector DBs can index new documents faster than LLM compilation
- **Structured queries** — Semantic search excels at finding specific passages across massive corpora

## Does Wiki Kill RAG?

As [[source-herk-llm-wiki-setup|Nate Herk]] concludes: "No, but kind of yes." It depends on scale and purpose:
- **Hundreds of pages with good indexes** → Wiki works great
- **Millions of documents** → Use traditional RAG

Karpathy himself noted: "I thought I had to reach for fancy RAG, but the LLM has been pretty good about auto-maintaining index files and brief summaries... at this ~small scale."

## Future Direction

Karpathy mentions the natural next step: "synthetic data generation + finetuning to have your LLM 'know' the data in its weights instead of just context windows." This would combine the organizational benefits of the wiki with weight-level knowledge.
